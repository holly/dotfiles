# git-commit スキル

コミット操作の承認フローと実行手順を定義する。
**このスキルは単独で動作しない。** 必ず以下を併用する：

- **`git-flow` スキル** — ブランチチェック・命名規則
- **`commitizen` スキル** — メッセージ形式

---

## 絶対ルール

1. **ユーザーの明示的な承認なしに `git commit` を実行しない**
2. **ユーザーの明示的な承認なしに `git push` を実行しない**
3. **`git-flow` スキルに従い、保護ブランチへの直接コミットを拒否する**
4. **コミットメッセージは `commitizen` スキルの形式に従う**

---

## ワークフロー

### ステップ0：ブランチ確認（最初に必ず実行）

```bash
git branch --show-current
```

`main` / `master` / `develop` だった場合 → **`git-flow` スキルの「保護ブランチ検出時の対応」に従い即座に止まる。**

### ステップ1：ステージ済み変更を確認する

```bash
git diff --staged
git status
```

差分を読んで変更内容を把握する。

### ステップ2：コミット範囲を判断する

- 複数の無関係な変更が混在 → 個別コミットへの分割を提案する
- まとまっている → 1コミットで進める

### ステップ3：コミットメッセージを作成する

`commitizen` スキルに従ってメッセージ案を作成する。

### ステップ4：承認を求める — **ここで必ず止まる**

以下の形式でユーザーに提示する：

```
以下の内容でコミットします。確認してください。

🌿 ブランチ: feature/oauth2-login
📁 ステージ済みファイル:
  modified: src/auth/login.ts
  modified: tests/auth.test.ts

📝 コミットメッセージ:
  feat(auth): add OAuth2 login support

よろしければ「はい」「OK」「進めて」などでお知らせください。
```

### ステップ5：コミットを実行する（承認後のみ）

```bash
git commit -m "<type>(<scope>): <subject>"
```

本文・フッターが必要な場合：
```bash
cat > /tmp/commit_msg.txt << 'EOF'
<type>(<scope>): <subject>

<本文>

<フッター>
EOF
git commit -F /tmp/commit_msg.txt
```

### ステップ6：完了確認

```bash
git log --oneline -3
```

結果を表示して終了。**`git push` は実行しない。**

---

## プッシュのポリシー

同じターンでユーザーが明示的に要求しない限り `git push` は実行しない。

「コミットしてプッシュして」と言われた場合でも：
1. コミット完了後に改めて確認する：「プッシュも実行しますか？」
2. 2度目の承認を得てからのみ実行する

---

## エッジケース

### ステージが空
```
ステージされているファイルがありません。
git add でファイルをステージしてから再度お知らせください。
```
ユーザーの指示なしに `git add` を実行しない。

### 未ステージの変更がある
```
以下のファイルに未ステージの変更があります：[一覧]
コミットに含めますか？
```

### 複数の関心事が混在
```
変更が複数の関心事にまたがっています。分割を推奨します：
1. feat(auth): ...
2. test(auth): ...
このまま1つにまとめますか？それとも分割しますか？
```
